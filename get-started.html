Getting started

{{Tell-Client}} has partnered with tell.connect to provide Third Party Providers (“TPPs”) with access to consumer account information and payment initiation in line with regulatory requirements.

Introduction

This guide outlines the process for registration, authorising and accessing consumer information as an AISP, and initiating payments as a PISP. The detailed API specification can be found in the spec section of the website.

The Specifications

Developer Registration

All TPPs must register on the {{Tell-Client}} developer portal. We use the principle of least based privilege access and will only perform full due diligence checks when a TPP requests production access.


Application Registration

TPPs may operate multiple applications with different requirements and security certification.

When you register an application on our portal, you will be issued with a unique "Client ID" & "Client Secret". For each application, you can provide a unique “Redirect URL” (the location we will return a user to after providing consent) and you have the option to upload an eIDAS certificate for authorisation and authentication.

The use of the Redirect URL, Client ID, Client Secret and eIDAS certificate are detailed below.

Environments

{{Tell-Client}} provides two separate environments:

Sandbox

The Sandbox environment is broadly identical to Production but rather than connecting to live systems, it provides stubbed data. Some of the tighter usage restrictions are also eased in Sandbox, such as the need for unique requests IDs (this is described in more detail below).

Production

The Production environment allows access to real client data. As you would expect this data is highly sensitive and we therefore are required to perform a higher level of due diligence and approval all requests for applications which would like to be moved into Production.

Account Access


In order to provide a TPP with a consumer’s data or to initiate a payment on their behalf, the consumer is required to grant permission to the TPP for a period not exceeding 90 days.

Initiate Consent

To begin the process, you should call the POST /authorise endpoint to obtain an intent-id. Please refer to the swagger file specification for details.

The data you need to provide includes:

interface An id associated with our interface – please use the following:
uuid A unique identifier which you should create to track a user’s authorisation end to end. In Production we will validate that the uuid is unique, and a request will be rejected if it is found to be duplicated. To assist your development, we do not enforce uniqueness in Sandbox.
scope Use “aisp” for account access or “pisp” for payment initiation
payinfo This is required if the scope is “pisp” i.e. the consent relates to a payment initation. You should provide a JSON Object specifying the payee/recipient, amount and other related information. Full details can be found in the API Spec.

This request requires Basic Authorization to be provided in the form of a Base64 encoded client_id and client_secret. You must also sign the request by including a valid JTW in the “x-jws-signature” header (see Message Signing below for more details)

You will receive two values in response:

  • - an intent-id - this is our reference for the request
  • - a location - this is where you should redirect the user for authentication

PSU Authorisation

Now, simply redirect the user to the location you received in the previous step.

{{Tell-Client}} will then guide the consumer through a consent journey which involves both user authentication and authorisation.

Authorisation Confirmation

In cases of both Pass & Fail when obtaining consent authorisation, we will redirect the consumer to the Redirect URL specified when you registered the application*.

* If you have not yet provided your own Redirect URL in the Sandbox environment, requests will be returned to our stubbed page, which presents a textual version of the response including the outcome and authorisation code, allowing you to manually proceed.

The response will also include the "uuid" value you created earlier in the process and the intend-id that we issued. This can be used to identify and cross reference the originating request for consent.

If the consent authorisation was successful, we will also include an authorisation code in the query parameters.

Redirect Response 

https://{{Your Redirect URL}}?intent-id={{intent-id}}&uuid={{uuid}}&authorisation_code={{Authorisation Code}}

Obtain Access Token

With the authorisation code returned, your application can now request an Access Token which can in turn be used to access the account resource information or initiate a payment.

The authorisation code is valid for 60 seconds, so your application must exchange it for an access token within that timeframe or the consent authorisation will need to be repeated.

This is done using the POST /access-token endpoint, providing the details as set out in the specification.

In the request, set grant_type to ”authorization_code” and code to the authorisation code received in the redirect.

This request requires Basic Authorization to be provided in the form of a Base64 encoded client_id and client_secret. You must also sign the request by including a valid JTW in the “x-jws-signature” header (see Message Signing below for more details)

The access token returned will permit access to the consumer data / payment initiation endpoint for a maximum of 1 hour at which point the access token will need to be refreshed.

Example Access Token Response

{
    "access_token": {{Your new valid access token}},
    "refresh_token": {{Your new valid refresh token}},
    "token_type": "Bearer",
    "expires_in": 3600,
    "scope": "balance:read,product:read,accounts:read,beneficiaries:read"
}
access_token An access token that can be used to make requests for this account holder / consent
refresh_token A refresh token that should be stored securely and used to refresh the access token when it expires
token_type Will always be “Bearer”
expires_in The duration in minutes that the access_token will be valid for (will typically be 3600)
scope This contains a set of permissions associated with the consent. It is a comma-separated list in the format: balance:read,product:read,accounts:read,beneficiaries:read

Refresh Access Token

When an access token has expired, you can utilise the stored refresh token to generate a new access token using the refresh token grant. Refresh tokens expire automatically after 90 days and new refresh tokens are issued after each use of the refresh token grant.

The API call is the same as for obtaining an access token described above, but with a grant_type of “refresh_code” and providing the refresh token in the code field.

Example Refresh Token Response

{
    "access_token": {{Your new valid access token}},
    "refresh_token": {{Your new valid refresh token}},
    "token_type": "Bearer", // Always "Bearer"
    "expires_in": 3600, // Default 1 hour. Can be adjusted on request
    "scope": "balance:read,product:read,accounts:read,beneficiaries:read"
}

Revoked Consent

A consumer may choose to revoke consent at any time. We do not proactively inform TPPs of revoked consents.

Account Access

With a valid consent and access token of type “aisp”, you can now start retrieving information from {{Tell-Client}} accounts.

Requests directly initiated by user activity (i.e. from your website or mobile application) must include the user’s IP address in a “x-user-ip” header. Requests initiated directly from your application (e.g. automated data sweeps, where the user has not manually instigated the task) should omit the “x-user-ip” header. Requests that are not initiated by the user are restricted to a maximum of 4 per endpoint per day.

These requests require Authorization in the form of a Bearer Token, which is the Access Token described above. You must also sign the request by including a valid JTW in the “x-jws-signature” header (see Message Signing below for more details)

Payment Initiation

With a valid consent and access token of type “pisp”, you can now submit a single payment initiation on behalf of the user.

These requests require Authorization in the form of a Bearer Token, which is the Access Token described above. You must also sign the request by including a valid JTW in the “x-jws-signature” header (see Message Signing below for more details)

Message Signing

All API calls must be secured with valid Authorization (see above for details) and must also be signed using a certificate that you upload and assign within the Portal.

Please ensure you have uploaded and registered your certificate against your application within the portal. When you register a certificate, you will be issued a unique Key ID – please keep this secure as it will be required for message signing.

You should construct a JWT token - see https://jwt.io/ for more detailed instructions. The specific format required is as follows:

Header:

{
"alg": "HS256",
"typ": "JWT",
"app": {{uuid of your app – this is available in the Portal}},
"path": {{endpoint e.g. "/authorise"}},
"iat": {{timestamp}}
}

Payload:

interface
scope {{aisp or pisp}}
uuid {{generate by your application}}

For instructions on how to sign and construct the full token, please refer to the JWT website or other information sources.

Going Live

When you’re ready to move into production, simply log into the portal, select the application and click “request go-live”. Our team will carry out a series of checks and may be in touch if we require any further information or testing, after which, we will release your application access into Production.
